1 動的化の理由

QuizGen は静的 HTML/CSS/JS として存在する。創業者承認（2026-05-29）を受けて以下4機能を解禁するための動的化設計を行う。

✓ DB に保存可能
🔐
API キー保護
✗ クライアントに露出 / BYOK
✓ 環境変数で完全隠蔽
🔗
短縮 URL
✗ ハッシュが 200〜400 文字
✓ /q/abc12345 でシェア
🖼️

✗ 静的 ogp.png 1枚のみ

継承して変更しない決定: ホスティング=Vercel、LLM=Groq Llama 3.3 70B（変更不可）、フロントエンド=既存 HTML/CSS/JS 流用

4 Groq API キー保護 & CORS
# 環境変数（Vercel ダッシュボード → Settings → Environment Variables）
GROQ_API_KEY=gsk_...
UPSTASH_REDIS_REST_URL=https://...
UPSTASH_REDIS_REST_TOKEN=...

// /api/generate.js — Node.js runtime + maxDuration:30 を推奨
// vercel.json: { "functions": { "api/generate.js": { "maxDuration": 30 } } }

const allowedOrigins = new Set([
  "https://quizgen.app",
  process.env.VERCEL_URL ? `https://${process.env.VERCEL_URL}` : "",
].filter(Boolean));

// CORS だけでは不十分 — Origin + Sec-Fetch-Site を両方検証
const origin = req.headers["origin"];
const secFetchSite = req.headers["sec-fetch-site"];
const crossSite = (origin && !allowedOrigins.has(origin)) ||
  (secFetchSite && !["same-origin","same-site","none"].includes(secFetchSite));
if (crossSite) return res.status(403).json({ error: "forbidden" });

// IP 別レート制限（10 req/IP/日）— @upstash/redis を使用
import { Redis } from "@upstash/redis";
const kv = Redis.fromEnv();

async function checkRateLimit(ip) {
  const key = `ratelimit:${ip}:${new Date().toISOString().slice(0,10)}`;
  const count = await kv.incr(key);
  if (count === 1) await kv.expire(key, 86400);
  return count <= 10;
}

⚠️ @vercel/kv は sunset済み。 Vercel Marketplace から Upstash Redis を追加し、@upstash/redis + Redis.fromEnv() を使用すること。

6 短縮 URL（Upstash Redis + nanoid）
選択肢コスト工数URL 例判定
Upstash Redis 自前実装$02〜3時間quizgen.app/q/abc12345採用
Bitly API無料枠1時間bit.ly/xxxxx却下（自ドメインでない）
TinyURL無料30分tinyurl.com/xxx却下（同上）

実装パターン

import { nanoid } from "nanoid";
import { Redis } from "@upstash/redis";
const kv = Redis.fromEnv();

// 8文字推奨（6文字は衝突リスク）— nx:true で衝突時リトライ（最大5回）
async function createShortId(quizJson) {
  if (JSON.stringify(quizJson).length > 16_384) throw new Error("quiz_too_large");
  for (let i = 0; i < 5; i++) {
    const id = nanoid(8);
    const ok = await kv.set(`quiz:${id}`, quizJson, { nx: true });
    if (ok) return id;
  }
  throw new Error("id_collision_retry_exhausted");
}
// シェア URL: https://quizgen.app/q/abc12345
// 既存ハッシュ URL (/#q2=...) は後方互換で永続動作

TTL: なしを推奨。1クイズ JSON ≈ 2KB、月500件 × 12ヶ月 = 6,000件/年。256MB 無料枠で実質20年分。KV 障害時は 404 ではなく 503 + 再試行 UI を返すこと。

7 動的 OGP（フェーズ2 — 設計変更済み）

フェーズ1.5 は静的 ogp.png を維持。月間シェアクリック 500回超でフェーズ2に移行。

URL 種別OGP 内容フェーズ
/q/:id クイズ共有クイズタイトル + 作成者名（固定画像）。score は含めない。Ph.1.5
/r/:attemptId 結果共有KV に保存した回答結果から og:image を生成。score を URL param で信用しない（改ざん防止）。Ph.2

⚠️ 設計変更: 旧設計の /api/og?score=4&total=5 は score の改ざんが可能 + SNS キャッシュ非対応のため廃止。フェーズ2では /r/:attemptId 方式を採用。

フェーズ2 実装設計（参考）

9 未解決事項 / architecture-reviewer 差し戻しポイント
9-1   短縮 URL の TTL 設計
90日 TTL vs TTLなし。Upstash 256MB で実質20年分保存可能。バイラルで SNS に貼られた URL の長期保持が必要かどうかはプロダクト判断。
architect 推奨: TTLなし。PII は payload に入れないこと。
9-2   Upstash Redis の可用性（SLA 99.9%）
KV 障害時にショートURL が機能しなくなるリスク。MVP では 503 + 再試行 UI で対処。重要導線なら Upstash paid / multi-region を検討。
architect 推奨: MVP は 503 fallback のみ。LocalStorage キャッシュはフェーズ2。
9-3   Functions タイムアウト設計
現行 Vercel は Node.js runtime で Hobby も vercel.json の maxDuration で 30秒指定可能。Edge Runtime は低レイテンシ/streaming が必要な場合のみ。
architect 推奨: Node.js runtime + maxDuration: 30。Edge は OGP 生成など明確なユースケースのみ使用。
9-4   ✓ 解決済み   Tally.so GDPR リスク
Tally.so は EU 企業でない → 誤り。Tally.so はベルギー/EU企業、GDPR compliant、DPA あり、データは Europe 保存。（Codex レビュー 2026-05-30）
残課題: プライバシーポリシーへの記載・削除依頼窓口・500人超での移行手順。法的ブロッカーではない。
9-5   ↻ 設計変更   動的 OGP と score の依存関係
/api/og?score=4&total=5 → score を query param で信用しない設計に変更。
新設計: /r/:attemptId で KV に保存した回答結果から og:image を生成（改ざん防止 + SNS キャッシュ対応）。
