1 動的化の理由

現状

Whoami Quiz は静的 HTML/CSS/JS として存在する。Vercel Functions による Groq API プロキシ設計が確定し、フェーズ1（バイラル検証）が進行中。 創業者承認（2026-05-29）を受けて以下4機能を解禁するための動的化設計を行う。

Vercel Hobby 採用根拠

  • 静的フロントと Functions が同一デプロイで完結（CI/CD コスト最小）
  • Vercel KV（Redis互換）・Vercel OG が同リポジトリで使える
  • 創業者がすでに利用経験のあるプラットフォーム
  • CLI（vercel コマンド）でデプロイ・ログ・env 管理が完結

4 Groq API キー保護
# Vercel ダッシュボード → Settings → Environment Variables
GROQ_API_KEY=gsk_...

# /api/generate.js — Edge Runtime で API キー隠蔽 + レート制限
export const config = { runtime: "edge" };  // 30秒タイムアウト（Hobby 10秒制限を回避）

// CORS: 同一オリジンのみ許可
const allowedOrigins = [
  "https://whoami-quiz.app",
  process.env.VERCEL_URL ? `https://${process.env.VERCEL_URL}` : null,
].filter(Boolean);

// IP 別レート制限 — Vercel KV で 10 req/IP/日
async function checkRateLimit(ip) {
  const key = `ratelimit:${ip}:${new Date().toISOString().slice(0,10)}`;
  const count = await kv.incr(key);
  if (count === 1) await kv.expire(key, 86400);
  return count <= 10;
}

Edge Runtime を使うと Vercel Hobby でも 30秒タイムアウトが使える（通常の Functions は 10秒）。Groq SDK との互換性は実装時に確認が必要（architecture-reviewer 9-3 参照）。

5 ウェイトリスト実装
選択肢コストセットアップデータ所有判定
Tally.so embed 無料 5分（iframe 1行） Tally 所有（CSV エクスポート可） 推奨 A
Vercel KV + 自前 API 無料 2〜3時間 完全自己所有 推奨 B（登録500人超で移行）
Resend Audience 無料枠（3,000件） 30分 Resend 所有 次善
Supabase Postgres 無料枠 1〜2時間 完全自己所有 オーバーキル（Ph.1）

移行トリガー（推奨A → 推奨B）: ウェイトリスト登録者 500人超、または登録後の自動メール送信・セグメント管理が必要になった場合。

フロー

// クイズ作成後: POST /api/shorten { quiz: JSONオブジェクト }
import { nanoid } from "nanoid";

const shortId = nanoid(6);  // 例: "abc123" — 560億通りで衝突ほぼゼロ
await kv.set(`quiz:${shortId}`, quizJSON);  // TTL なし推奨
return { shortId };

// シェア URL: https://whoami-quiz.app/q/abc123
// 既存ハッシュ URL (/#q2=...) は後方互換で永続動作

TTL: なしを推奨。1クイズ JSON ≈ 2KB。月500件 × 12ヶ月 = 6,000件/年。256MB 無料枠で実質 20年分保存可能。

// GET /api/og?id=abc123&score=4&total=5 — Vercel OG（@vercel/og）
import { ImageResponse } from "@vercel/og";
export const config = { runtime: "edge" };

export default function handler(req) {
  const { score, total } = Object.fromEntries(new URL(req.url).searchParams);
  return new ImageResponse(
    <div style={{ background: "#0f1117", width: "1200px", height: "630px",
                  display: "flex", flexDirection: "column",
                  alignItems: "center", justifyContent: "center", color: "#fff" }}>
      <div style={{ fontSize: 80, fontWeight: 700 }}>{score}/{total}</div>
      <div style={{ fontSize: 36, color: "#00d4b8" }}>Whoami Quiz</div>
    </div>,
    { width: 1200, height: 630 }
  );
}

9 未解決事項（architecture-reviewer 差し戻しポイント）
9-1　短縮 URL の TTL 設計妥当性
現行設定 90日 TTL → 変更案はなし（256MB 無料枠で実質20年分保存可能）
architect 推奨: TTL なし。architecture-reviewer の確認求む。
9-2　Vercel KV の可用性（SLA 99.9%）
KV 障害時にショート URL が 404 になるリスク。代替: LocalStorage キャッシュ（フェーズ2）。
architect 推奨: MVP では KV のみで十分。
9-3　Vercel Hobby Functions タイムアウト（10秒）
Groq Llama 3.3 70B はピーク時に 10秒に近づく可能性。Edge Runtime（30秒）で回避できるが Groq SDK 互換要確認。
architect 推奨: まず Edge Runtime で実装し互換性を確認。
9-4　ウェイトリスト（Tally.so）GDPR リスク
Tally.so は EU 企業でないため、日本語ユーザーのメアドを外部に保存することの法的リスク。
architecture-reviewer に法的リスク評価を求む。
9-5　動的 OGP とスコアの依存関係（UX 判断）
og:image URL にスコアを含める設計（/api/og?score=4&total=5）は回答者ごとに OGP が変わる。代替: OGP はクイズタイトルのみ固定。
フェーズ2着手前に PM と確認が必要。
